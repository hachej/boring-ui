# Ask User Questions Plugin Spec

Last updated: 2026-05-08

## Summary

Add a blocking `ask_user` capability for agent sessions that opens a Workspace
panel named **Questions**, renders an agent-generated form, lets the user submit
answers, and returns those answers to the waiting agent tool call.

Workspace owns the `askUserPlugin` UI, form primitives, persistence-facing
contracts, and UI bridge command handling. Agent owns the blocking tool runtime
and session-level waiter. Browser answers are dispatched through a narrow Questions browser -> server
command channel. Existing `UiBridge.postCommand` remains the agent/server -> UI
command path for opening/focusing the Questions pane.

## Goals

- Provide an agent tool that asks the user for structured input and blocks until
  the user answers, cancels, times out, or the tool is aborted.
- Open or focus a Workspace panel titled `Questions` whenever a question becomes
  pending.
- Render forms from a portable, JSON-serializable schema generated by the agent.
- Allow the Questions pane to open immediately in draft mode and display form
  content progressively as the agent streams form patches, before the final full
  schema is complete.
- Provide primitive React UI components and hooks so app shells can render these
  forms in a standard way.
- Persist the single pending question per agent session across browser reloads.
- Use the existing UI bridge direction for agent/server -> UI commands and add
  a narrow Questions browser -> server command channel for submit/cancel.
- Keep Workspace base front/shared code package-neutral; no value imports from
  `@boring/agent` outside allowed app composition boundaries.

## Non-goals

- Multiple simultaneous pending questions per agent session.
- File upload fields in the first version.
- Conditional/dependent fields such as "show field B only if A === X". V1 forms are flat.
- Localization/i18n of labels. V1 renders strings as supplied.
- Full process-crash restoration of an already-blocked in-memory tool call.
- Using chat-stream `data-ui-command` parts as the source of truth for answers.
- Making `pi-ask-user` the primary Workspace UX implementation dependency.
- Isolation-specific exclusions. `ask_user` should work in every supported
  agent execution mode (`direct`, `local`, and `vercel-sandbox`) through the
  active runtime-mode adapter.

## User flow

1. Agent calls `ask_user` with either a complete title/context/form schema or a
   streaming form builder.
2. Server generates a globally unique `questionId` and creates a pending question
   record for the current agent session. Streaming requests start as `draft`;
   complete-schema requests can start as `ready` after validation.
3. Agent tool blocks on an `AskUserCoordinator` waiter keyed by `questionId`.
4. Workspace receives/loads the pending question and opens or focuses the
   `Questions` panel immediately. Draft questions render skeleton/progressive
   fields; ready questions render the submittable form.
5. For streaming requests, agent/tool form patches update the draft until a
   `finalize` patch validates the full schema and marks the question `ready`.
6. User fills the form and clicks Submit. Submit is disabled until `ready`.
7. Questions panel calls the Questions browser -> server command channel with
   `{ kind: "questions.submit", params: { ... } }`.
8. The Questions command handler accepts or rejects the command and returns only
   command status to the browser.
9. Workspace server validates the command, stores the answer, and resolves the
   matching `AskUserCoordinator` waiter.
10. Tool promise resolves with `AskUserToolResult`; the result returns to the
   agent runtime, not to the browser command response.

Cancel follows the same path with `kind: "questions.cancel"` and resolves the
waiter with `status: "cancelled"` and `reason: "user_cancelled"`.

## Package placement

### `@boring/workspace/plugins/askUserPlugin/shared`

The ask-user contracts are plugin-owned and live under the dedicated plugin
shared module, not generic `@boring/workspace/shared`. Keep the plugin
self-contained:

- `shared/types.ts` exports `AskUserQuestion`, `AskUserRequest`,
  `AskUserFormSchema`, `AskUserField`, `AskUserAnswer`,
  `QuestionsSubmitCommand`, and `QuestionsCancelCommand`.
- `shared/error-codes.ts` exports ask-user-specific stable error codes.
- `shared/schema.ts` exports browser-safe runtime validators and inferred types
  for request/schema/patch/answer payloads.
- `shared/constants.ts` exports plugin ids, surface kinds, command kinds, limits,
  and UI-state slot names.

No `node:*`, no `Buffer`, no `@boring/agent` value imports. Plugin shared code
may import only from published generic Workspace shared subpaths/barrels (for
example surface or UI-command types that are part of the public Workspace SDK),
not from internal Workspace implementation files. Generic Workspace shared code
must not import ask-user plugin contracts.

Only re-export ask-user contracts from a public subpath when intentionally making
them part of the supported plugin API, e.g. `@boring/workspace/ask-user` or a
plugin-specific export path. Do not add them to the root Workspace shared barrel
by default.

Workspace must not import agent error-code enums to satisfy stable-code needs.
The ask-user plugin owns a plugin-scoped canonical error-code enum in
`shared/error-codes.ts`; all ask-user front/server/shared code imports codes from
that one file and never uses raw string codes. Invariant lint should treat this
file as the canonical import site for ask-user plugin errors. The enum includes
at least:

- `ASK_USER_PENDING_EXISTS`
- `ASK_USER_QUESTION_NOT_FOUND`
- `ASK_USER_QUESTION_ABANDONED`
- `ASK_USER_SCHEMA_INVALID`
- `ASK_USER_ANSWER_INVALID`
- `ASK_USER_SESSION_MISMATCH`
- `ASK_USER_UI_UNAVAILABLE`
- `ASK_USER_PATCH_INVALID`
- `ASK_USER_PATCH_STALE`
- `ASK_USER_ALREADY_ANSWERED`
- `ASK_USER_ALREADY_CANCELLED`
- `ASK_USER_UNAUTHORIZED`
- `ASK_USER_UI_ACK_TIMEOUT`
- `ASK_USER_RATE_LIMITED`
- `ASK_USER_RUNTIME_UNAVAILABLE`

### `@boring/workspace/plugins/askUserPlugin`

New default Workspace plugin. The package name follows the capability
(`askUserPlugin`); the user-facing panel title is **Questions**.

The plugin owns all ask-user implementation details:

- `front/QuestionsPanel.tsx`
- `front/PendingQuestionsWatcher.tsx`
- `front/usePendingQuestion.ts`
- `front/primitives/QuestionForm.tsx`
- `front/primitives/QuestionField.tsx`
- field primitive components
- `server/questionsBridge.ts`
- `shared/types.ts`
- `shared/schema.ts`
- `shared/error-codes.ts`
- `shared/constants.ts`

Front plugin registration should follow the plugin-agent-layer authoring
shape. Prefer a factory captured through the public plugin authoring API (or the
`boringFrontFactoryToPlugin` helper where that is the active API), registering:

- `panels.register(...)` / `registerPanel(...)` for the `Questions` panel
- `surfaceResolvers.register(...)` / `registerSurfaceResolver(...)` for
  `openSurface({ kind: "questions", target })`
- `commands.register(...)` / `registerPanelCommand(...)` for an optional
  palette/menu action named `Open Questions`
- `providers.register(...)` / `registerProvider(...)` for
  `PendingQuestionsWatcher` when provider outputs are enabled by the active
  plugin-agent-layer branch

The resulting Workspace front plugin outputs are the existing front kinds:
`panel`, `surface-resolver`, optional `command`, and optional `provider`. There
is no separate `page` output kind. The ask-user tool is not contributed through a
front `agent-tool` output; it is contributed by the ask-user server plugin as a
Pi extension factory so the server-side runtime dependencies stay plugin-owned.

Server-side bridge handling belongs to a server plugin fragment using
`defineServerPlugin()`, not a front plugin output kind. That fragment registers
the browser -> server Questions routes, provisions the
`AskUserStore`/`AskUserCoordinator` runtime, and contributes the `ask_user` tool
through a plugin-owned Pi extension factory.

Intended shape:

```ts
export const askUserServerPlugin = defineServerPlugin({
  id: ASK_USER_PLUGIN_ID,
  label: "Ask User",
  routes: questionsRoutes,
  extensionFactories: [createAskUserPiExtensionFactory({ runtime })],
})
```

`questionsRoutes` mounts the `questions.submit`, `questions.cancel`, and
`questions.opened` browser -> server command channel on the Workspace Fastify app
with the app's normal authenticated request context and CSRF/origin protections.
`createAskUserPiExtensionFactory()` registers the `ask_user` tool with Pi and
closes over the plugin-owned runtime; it does not require ask-user exports from
`@boring/agent/server`.

### `@boring/workspace/app/*`

App composition wires `askUserPlugin` into the generic Workspace + Agent server
composition. This is the allowed boundary for Workspace + Agent product
composition. App code imports ask-user contracts from the plugin/public ask-user
subpath, not from generic Workspace shared barrels.

### `@boring/agent/server`

No ask-user-specific exports are required from `@boring/agent/server`. Agent
server only needs to keep supporting generic plugin-contributed tools/extensions
and runtime dependencies. Ask-user-specific factories, stores, coordinators, and
command handlers stay inside `askUserPlugin`.

### `@boring/workspace/plugins/askUserPlugin/server`

The ask-user plugin owns the blocking `ask_user` tool factory and runtime
interfaces. These do not belong to `@boring/agent/server` as ask-user-specific
exports.

The plugin server module owns:

- `server/createAskUserTool.ts`
- `server/createAskUserRuntime.ts`
- `server/AskUserCoordinator.ts`
- `server/AskUserStore.ts`
- `server/questionsBridge.ts`
- `server/questionsRoutes.ts`
- `server/createAskUserPiExtensionFactory.ts`
- `server/askUserServerPlugin.ts`

It exposes:

```ts
type AskUserRequest = {
  sessionId: string
  title?: string
  context?: string
  schema?: AskUserFormSchema
  timeoutMs?: number
}

type AskUserToolInput = {
  title: string
  context?: string
  schema: AskUserFormSchema
  timeoutMs?: number
}

type AskUserPatchResult =
  | { ok: true; question: AskUserQuestion }
  | { ok: false; code: string; message: string; currentVersion?: number }

interface AskUserStreamHandle {
  questionId: string
  patch(patch: AskUserFormPatch, options?: { expectedVersion?: number }): Promise<AskUserPatchResult>
  finalize(options?: { submitLabel?: string; expectedVersion?: number }): Promise<AskUserPatchResult>
  wait(options?: { signal?: AbortSignal }): Promise<AskUserToolResult>
  cancel(reason: AskUserCancelReason): Promise<AskUserToolResult>
}

interface AskUserCoordinator {
  registerWaiter(questionId: string, sessionId: string, signal?: AbortSignal): Promise<AskUserToolResult>
  resolveAnswered(questionId: string, answer: AskUserAnswer): Promise<boolean>
  resolveCancelled(questionId: string, reason: AskUserCancelReason): Promise<boolean>
}

interface AskUserRuntime {
  askUser(request: AskUserRequest & { schema: AskUserFormSchema }, options?: { signal?: AbortSignal }): Promise<AskUserToolResult>
  beginAskUserStream(request: Omit<AskUserRequest, "schema">, options?: { signal?: AbortSignal }): Promise<AskUserStreamHandle>
}
```

The LLM-facing tool surface is a single tool named `ask_user` with
`AskUserToolInput` JSON. Streaming is not a second LLM-visible tool. The harness
may feed partial tool-argument JSON into `beginAskUserStream()` to render draft
fields progressively, then call `finalize()` when the JSON parser commits. If the
harness cannot stream partial tool arguments, it falls back to `askUser()` after
receiving the complete JSON.

`AskUserCoordinator` has a default in-process implementation in v1. Multi-process
or cloud deployments can later inject a pub/sub-backed coordinator without
changing tool, store, or plugin callers.

The ask-user server plugin contributes the tool to the agent through
`extensionFactories: [createAskUserPiExtensionFactory({ runtime })]`. The Pi
extension factory registers the `ask_user` tool and uses the plugin-owned
runtime. It should not require `@boring/agent/server` to import or export
ask-user-specific symbols.

Workspace app/server composition injects or constructs a host-side runtime that
stores the pending question and resolves through UI bridge answers. The
`ask_user` tool executes in the host Pi/runtime process for every supported
execution mode. Runtime modes (`direct`, `local`, `vercel-sandbox`) affect the
workspace/filesystem/sandbox substrate; they do not require a sandbox -> host
ask-user callback in v1. If the host ask-user runtime is not registered, app boot
or capability negotiation should fail fast with `ASK_USER_RUNTIME_UNAVAILABLE`
rather than hanging a tool call.

If `options.signal` aborts
because the run/session is cancelled, the runtime must clear or cancel the
pending question and resolve the tool with `status: "cancelled"` and
`reason: "aborted"`. If `timeoutMs` elapses first, resolve with
`reason: "timeout"`.

## Form schema

Use a small JSON-schema-lite wire contract. It is serializable and easy to
render in React. Validation may use Zod or equivalent schemas derived from this
field union on both client and server; the lite shape is the transport contract,
not a ban on runtime validators.

Complete-schema mode validates the whole schema before creating a `ready`
question. Streaming mode validates each patch quickly, stores a draft question,
and validates the assembled schema on `finalize` before enabling submit. Tool
boundary validation must fail fast with actionable errors before any complete
schema blocks the agent.

```ts
type AskUserFormSchema = {
  wireVersion: 1
  fields: AskUserField[]
  submitLabel?: string
}

const ASK_USER_SCHEMA_LIMITS = {
  maxFields: 8,
  maxOptionsPerField: 50,
  maxFieldNameLength: 64,
  maxTitleLength: 200,
  maxLabelLength: 160,
  maxHelpTextLength: 500,
  maxContextLength: 4000,
  maxSerializedSchemaBytes: 32_000,
  maxFreeformAnswerLength: 4_000,
  minTimeoutMs: 1_000,
  maxTimeoutMs: 30 * 60_000,
  defaultTimeoutMs: 10 * 60_000,
}

type AskUserField =
  | {
      type: "text"
      name: string
      label: string
      required?: boolean
      placeholder?: string
      defaultValue?: string
      helpText?: string
      minLength?: number
      maxLength?: number
      pattern?: string
    }
  | {
      type: "textarea"
      name: string
      label: string
      required?: boolean
      placeholder?: string
      defaultValue?: string
      helpText?: string
      minLength?: number
      maxLength?: number
    }
  | {
      type: "select"
      name: string
      label: string
      required?: boolean
      options: AskUserOption[]
      defaultValue?: string
      helpText?: string
    }
  | {
      type: "multiselect"
      name: string
      label: string
      required?: boolean
      options: AskUserOption[]
      defaultValue?: string[]
      helpText?: string
      minSelections?: number
      maxSelections?: number
    }
  | {
      type: "checkbox"
      name: string
      label: string
      defaultValue?: boolean
      helpText?: string
    }
  | {
      type: "radio"
      name: string
      label: string
      required?: boolean
      options: AskUserOption[]
      defaultValue?: string
      helpText?: string
    }
  | {
      type: "number"
      name: string
      label: string
      required?: boolean
      defaultValue?: number
      placeholder?: string
      helpText?: string
      min?: number
      max?: number
      step?: number
      integer?: boolean
    }

type AskUserOption = {
  value: string
  label: string
  description?: string
}
```

Unknown future field `type` values must degrade gracefully in the default UI
with an unsupported-field message rather than crashing. Future additive field
types such as `date` are allowed as non-breaking schema extensions.

Validation and submission rules:

- `wireVersion` must be `1`; unknown versions fail with `ASK_USER_SCHEMA_INVALID`.
- `timeoutMs` must respect `ASK_USER_SCHEMA_LIMITS.minTimeoutMs` and
  `maxTimeoutMs`; omitted timeout uses `defaultTimeoutMs`.
- Field names must match `^[A-Za-z][A-Za-z0-9_-]{0,63}$` and must not be
  `__proto__`, `prototype`, or `constructor`.
- Field names must be unique within the form.
- Option values must be unique within a field.
- Select/radio fields require at least two options; multiselect requires at
  least one option.
- Labels, title, help text, context, field count, option count, and serialized schema
  size must respect `ASK_USER_SCHEMA_LIMITS`.
- Unknown keys in schema/field/option objects should be rejected or stripped
  consistently before persistence.
- Defaults must match the field type and, for option fields, must reference
  existing option values.
- Required validation runs server-side against the immutable finalized schema
  stored on the `AskUserQuestion` record. Complete-schema mode stores it at
  creation; streaming mode stores it at finalize.
- Answer values should be stored as a null-prototype map or array-of-entries
  internally to avoid prototype pollution hazards from generated field names.
- `number` values are finite JS numbers. `NaN` and `Infinity` are rejected;
  `integer: true` requires a finite integer.
- Text/textarea constraints must be internally consistent; `pattern` must be
  RE2-safe or otherwise guaranteed not to cause server-side ReDoS.
- Multiselect constraints must be internally consistent and bounded by option count.
- `checkbox` values are always boolean.
- Missing optional text/textarea/select/radio values normalize to `null`.
- Missing optional multiselect values normalize to `[]`.
- Submitted options must match the stored field option values.

Question and answer shapes:

```ts
type AskUserQuestionStatus = "draft" | "ready" | "answered" | "cancelled" | "abandoned"

type AskUserQuestion = {
  questionId: string
  sessionId: string
  ownerPrincipalId: string
  status: AskUserQuestionStatus
  title?: string
  context?: string
  draftFields?: AskUserField[]
  schema?: AskUserFormSchema
  draftVersion: number
  answerToken: string
  createdAt: string
  updatedAt: string
}

type AskUserAnswer = {
  questionId: string
  sessionId: string
  values: Record<string, string | string[] | boolean | number | null>
  submittedAt: string
}
```

Tool result shape:

```ts
type AskUserCancelReason = "user_cancelled" | "timeout" | "aborted" | "ui_unavailable" | "abandoned" | "rate_limited" | "runtime_unavailable"

type AskUserToolResult =
  | { status: "answered"; answer: AskUserAnswer }
  | {
      status: "cancelled"
      questionId: string
      sessionId: string
      reason: AskUserCancelReason
    }
```

`ownerPrincipalId` is set server-side when the pending question is created. In
standalone/no-auth v1 deployments it may use a fixed sentinel such as
`"anonymous"`; core/cloud deployments inject the real authenticated principal
from the request/session context.

## Progressive form streaming

The Questions pane may display the form progressively while the agent is still
constructing it. This is pane streaming, not inline chat rendering.

Streaming is represented as server-validated patches against the pending
question draft:

```ts
type AskUserFormPatch =
  | ({ patchId: string } & { type: "set_title"; title: string })
  | ({ patchId: string } & { type: "set_context"; context: string })
  | ({ patchId: string } & { type: "add_field"; field: AskUserField })
  | ({ patchId: string } & { type: "update_field"; name: string; patch: Partial<AskUserField> })
  | ({ patchId: string } & { type: "remove_field"; name: string })
  | ({ patchId: string } & { type: "finalize"; submitLabel?: string })
```

Patch rules:

- Creating a streaming question opens the Questions panel immediately in
  `draft` state.
- `patchId` is a caller-generated idempotency key. Reapplying the same patch id
  for the same question is a no-op that returns current state without advancing
  `draftVersion`; stores retain the last N patch ids per question.
- Patch application is ordered per `questionId` and guarded by monotonically
  increasing `draftVersion` / `expectedVersion` compare-and-swap semantics.
- Each patch gets fast validation and actionable errors for the agent.
- `add_field` fails on duplicate field names.
- `update_field` and `remove_field` fail for unknown field names.
- `update_field.patch.type` is forbidden. Changing field type requires
  remove/add so incompatible defaults/options cannot survive a type mutation.
- Patches are rejected once status is not `draft`.
- `finalize` is one atomic store operation: assemble `{ fields, submitLabel }`,
  run full schema validation, store immutable `schema`, and flip `draft -> ready`
  only if the expected draft version still matches.
- User submit is disabled until status is `ready`.
- If the agent aborts, times out, or sends invalid patches it cannot repair, the
  question resolves as cancelled with the appropriate reason.

UI behavior while streaming:

- Show title/context as soon as available.
- Show skeleton content before the first field.
- Render each valid field as it arrives, but keep the submit button disabled and
  labelled as waiting for the agent until finalization.
- When final validation succeeds, enable the form.
- If final validation fails, keep the pane in draft/error state and surface the
  validation error to the agent so it can patch and finalize again.

Complete-schema mode is still supported for simple cases and should internally
behave like creating a ready question from a validated assembled schema.

## Transcript and session reconstruction

Every `ask_user` interaction should produce a structured session event log so
the agent transcript, replay UI, and debugging tools can reconstruct what
happened without relying on ephemeral browser state.

The ask-user plugin store owns this event log. V1 ships a file-backed adapter;
core/cloud can inject a DB-backed adapter without changing plugin callers.
Prefer append-only transcript events over mutating one transcript record, because
append-only events are easier to replay and debug.

Record at least:

```ts
type AskUserTranscriptEvent =
  | {
      type: "created"
      question: AskUserQuestion
      at: string
    }
  | {
      type: "patched"
      questionId: string
      sessionId: string
      patch: AskUserFormPatch
      draftVersion: number
      at: string
    }
  | {
      type: "ready"
      questionId: string
      sessionId: string
      schema: AskUserFormSchema
      at: string
    }
  | {
      type: "answered"
      answer: AskUserAnswer
      at: string
    }
  | {
      type: "cancelled"
      questionId: string
      sessionId: string
      reason: AskUserCancelReason
      at: string
    }
  | {
      type: "abandoned"
      questionId: string
      sessionId: string
      at: string
    }
```

Rules:

- Events use server-stored question schema/draft, never a client-supplied schema.
- The transcript view derives read-only display state by replaying events.
- The transcript view may render a compact read-only summary after completion,
  but the active form is not displayed inline in chat.
- Transcript events are for reconstruction/debugging only. They are not the
  source of truth for accepting browser answers; the live submit path remains the
  Questions browser -> server command channel -> `AskUserCoordinator`.

## Session binding

The agent/session runtime supplies `sessionId` when it creates `AskUserRequest`.
The Questions front end obtains the active `sessionId` from the same agent
session context used by the app shell to render the active chat/session.

Routing is by server-generated, globally unique `questionId`. `sessionId` in
commands is validated for authorization and mismatch detection; it is not the
primary coordinator lookup key.

If multiple sessions share one Workspace process, the default in-process
`AskUserCoordinator` stores waiters in a process-local map keyed by `questionId`,
with each waiter carrying its expected `sessionId`. The coordinator interface is
injectable so future multi-process deployments can use Redis, Postgres
`LISTEN/NOTIFY`, or another pub/sub mechanism without changing callers.

## Persistence and orphan handling

Only one pending question is allowed per agent session. A second
`createPending()` for the same session must throw `ASK_USER_PENDING_EXISTS`; this
should be unreachable under normal blocking agent execution, but protects
against concurrent tools or duplicated requests.

Store interface:

```ts
interface AskUserStore {
  getPending(sessionId: string): Promise<AskUserQuestion | null>
  getByQuestionId(questionId: string): Promise<AskUserQuestion | null>
  createPending(question: AskUserQuestion): Promise<void>
  applyPatch(questionId: string, patch: AskUserFormPatch, expectedVersion?: number): Promise<AskUserQuestion>
  finalize(questionId: string, submitLabel?: string, expectedVersion?: number): Promise<AskUserQuestion>
  answer(questionId: string, answer: AskUserAnswer): Promise<void>
  cancel(questionId: string): Promise<void>
  markAbandoned(questionId: string): Promise<void>
  clearPending(sessionId: string): Promise<void>
  appendTranscriptEvent(event: AskUserTranscriptEvent): Promise<void>
  listTranscriptEvents(sessionId: string): Promise<AskUserTranscriptEvent[]>
  getTranscriptEventsForQuestion(questionId: string): Promise<AskUserTranscriptEvent[]>
}
```

Initial implementation should provide a file-backed or existing workspace-store
adapter. Core/cloud can later inject a DB-backed implementation. `AskUserStore`
owns both pending state and transcript event persistence for the plugin;
runtime/route handlers call `appendTranscriptEvent()` when creating, patching,
finalizing, answering, cancelling, or abandoning a question. Replay/debug UIs use
`listTranscriptEvents()` or `getTranscriptEventsForQuestion()`.

The store MUST emit a change event after every mutation (`createPending`,
`applyPatch`, `finalize`, `answer`, `cancel`, `markAbandoned`, `clearPending`) so
the UI state publisher can republish `questions.pending` for the affected
session. The Questions panel subscribes to that state and treats it as the source
of truth for initial render, draft patches, status transitions, and terminal
outcomes. `openSurface` is fire-and-focus only; it never carries patch payloads.

The runtime MUST apply configurable per-session and per-principal rate limits on
`createPending` (suggested defaults: 6 per rolling 60s per session, 30 per
rolling hour per principal). Rate limiting resolves the tool as cancelled with
`reason: "rate_limited"` and stable code `ASK_USER_RATE_LIMITED`.

The runtime SHOULD emit operational events through a no-op-by-default event sink:
`ask_user.created`, `ask_user.finalized`, `ask_user.answered`,
`ask_user.cancelled`, `ask_user.abandoned`, and `ask_user.ui_unavailable`.
Events must not include answer values.

Persistence requirement for v1: pending question survives browser reload and the
Questions panel can re-render it. Full server-process crash resumption of the
blocked tool waiter is a non-goal.

If a pending question exists but no waiter is registered for its `questionId`,
the server must treat it as abandoned: mark it abandoned, clear the active
pending state for the session, and return `ASK_USER_QUESTION_ABANDONED` through
the bridge command result. If the waiter is still alive when abandonment is
detected, resolve it with `status: "cancelled"` and `reason: "abandoned"`. On
server startup, the runtime should mark persisted `pending` records as abandoned
unless the agent session runtime explicitly rehydrates matching waiters.

If Workspace mode cannot reach any UI capable of rendering the Questions panel,
the runtime must fail deterministically with `ASK_USER_UI_UNAVAILABLE` and
resolve the tool with `status: "cancelled"` and `reason: "ui_unavailable"`.
Standalone CLI/TUI mode may implement its own native ask-user adapter using
`pi-ask-user` only as inspiration; do not depend on or wrap `pi-ask-user` as-is.

## Bridge and command channels

Do not make the whole `UiBridge` bidirectional in v1. Preserve the current
bridge direction:

- `UiBridge.postCommand` / `openSurface` remain agent/server -> UI commands.
  `askUserPlugin` uses this path to open or focus the Questions pane.
- Browser submit/cancel/opened uses a narrow Questions browser -> server command
  channel owned by the Questions server plugin. The underlying transport may be
  an HTTP route, but conceptually it is not the generic UI command queue.

Submit command shape:

```ts
questionsClient.dispatch({
  kind: "questions.submit",
  params: {
    questionId,
    sessionId,
    answerToken,
    values,
  },
})
```

Cancel:

```ts
questionsClient.dispatch({
  kind: "questions.cancel",
  params: {
    questionId,
    sessionId,
    answerToken,
  },
})
```

Opened acknowledgement:

```ts
questionsClient.dispatch({
  kind: "questions.opened",
  params: { questionId, sessionId },
})
```

The browser command channel returns only command acceptance state, e.g.
`{ status, error? }`. It does not return `AskUserToolResult`. The result returns
to the agent when the server-side Questions command handler resolves the matching
coordinator waiter.

Questions command handlers must:

- validate `kind` and `params`
- derive authorized user/workspace/session from authenticated request context;
  never trust browser-supplied `sessionId` alone
- validate `sessionId`, `questionId`, and `answerToken` against the stored
  question using constant-time token comparison
- validate submitted values against the immutable stored form schema
- enforce status transitions: only `ready -> answered` or `ready -> cancelled`
  can resolve the waiter
- handle retries idempotently: duplicate same submit may return the prior
  accepted result; submit after cancel returns `ASK_USER_ALREADY_CANCELLED`;
  cancel after answer returns `ASK_USER_ALREADY_ANSWERED`
- resolve concurrent submits deterministically: the first submit that wins the
  `ready -> answered` compare-and-swap is accepted; later conflicting submits
  return `ASK_USER_ALREADY_ANSWERED` and the losing tab renders the accepted
  answer from published state
- persist answer/cancel state
- resolve the matching `AskUserCoordinator` waiter exactly once
- emit/update Workspace state so the Questions panel leaves pending mode

### UI availability and open acknowledgement

Do not rely only on a coarse heartbeat. The Questions panel posts a per-question
`questions.opened` acknowledgement when it successfully mounts for a given
`questionId`. Before blocking indefinitely, the runtime should:

1. create the pending question
2. send `openSurface({ kind: "questions", target: questionId })` through the
   existing server -> UI bridge
3. wait for matching `questions.opened` with bounded timeout
   (`askUserOpenAckTimeoutMs`, default 8000ms, min 1000ms, max 60000ms)
4. continue waiting when the ack appears, or cancel with
   `ASK_USER_UI_UNAVAILABLE` / `reason: "ui_unavailable"` after timeout

A disconnected browser does not necessarily make persisted draft/ready state
invalid, but v1 should not leave a new blocking tool call waiting forever when no
Questions-capable UI acknowledges the open request. Reload counts as a remount
and re-emits `questions.opened` for the rehydrated `questionId`.

Security requirements:

- `answerToken` is a cryptographically random opaque string with at least 128
  bits of entropy, generated at `createPending`, and never reused.
- Browser -> server commands are protected by Workspace's authenticated request
  context plus CSRF/origin protection for state-changing commands.
- Command handlers verify authenticated principal equals `ownerPrincipalId`, the
  session matches the stored question, the token matches where required, and the
  status transition is valid before accepting values.

## Chat and Questions panel behavior

The active form is displayed in the Workspace workbench pane, not inline in the
chat transcript. Chat/tool rendering may show only a compact status card such as
"Waiting for your answer in Questions" with an action to open/focus the panel.
Any streamed chat data is display-only and must not become the answer source of
truth.

Chat/tool status lifecycle should be explicit and compact:

- `draft`: "Preparing question in Questions..."
- `ready`: "Waiting for your answer in Questions..."
- `answered`: compact read-only answer summary
- `cancelled`: cancelled with reason
- `timeout` / `abandoned` / `ui_unavailable`: closed with stable reason

- Register a panel titled `Questions`.
- A `PendingQuestionsWatcher` provider subscribes to pending-question state. On
  transition from no pending question to pending question, it opens a questions
  surface, e.g. `openSurface({ kind: "questions", target: questionId })`.
- The questions `surface-resolver` maps that surface to the concrete Questions
  panel.
- Show title, context, form fields, submit button, and cancel affordance.
- For draft streaming questions, show progressive fields and skeleton/loading
  state while keeping submit disabled.
- Disable submit while status is not `ready`, validation fails, or command is in
  flight.
- If the user closes the Questions pane while a question is pending, this does
  not cancel the tool call. The pending question remains in store and the chat
  status card continues to show "Waiting for your answer in Questions" with an
  Open Questions action. Re-opening the pane rehydrates the same draft/ready
  form and posts a fresh `questions.opened` ack.
- Explicit cancel requires the pane cancel affordance or an equivalent command,
  and should confirm when the form is dirty.
- After terminal status (`answered`, `cancelled`, `abandoned`, `timeout`, or
  `ui_unavailable`), show a compact terminal-state card for
  `answeredDisplayMs` (default 5000ms), then transition to the empty state. The
  pane does not auto-close.
- If no pending question exists, show an empty state: "No questions right now."

## Primitive UI package

Expose primitives from Workspace front code so apps can customize the page while
keeping standard schema behavior:

- `<QuestionFormProvider question state errors onSubmit onCancel />`
- `<QuestionForm schema values onValuesChange onSubmit onCancel disabled draft submitting errors />`
- `<QuestionFields fields rendererRegistry renderUnsupportedField />`
- `<QuestionField field value onChange error disabled />`
- `<QuestionSubmitButton disabled submitting draft />`
- `<QuestionCancelButton disabled />`
- `useQuestionForm(schema, initialValues?)`
- `validateQuestionValues(schema, values): QuestionValidationResult`
- `QuestionFieldRendererRegistry` for app-shell field overrides
- `QuestionFormState` and `QuestionValidationResult` types

Primitives should be headless-enough to style, but the default Questions panel
should provide a polished complete UI.

Default panel accessibility and UX requirements:

- announce new questions and terminal outcomes through an ARIA live region
- focus the first focusable field when status becomes `ready`, and focus the
  first invalid field after failed submit
- support Cmd/Ctrl+Enter to submit when valid
- support Escape to trigger cancel, with confirmation when values are dirty
- connect labels, help text, and validation errors with `aria-describedby`,
  `aria-invalid`, and `aria-errormessage`
- do not trap focus globally; this is a workbench pane, not a modal
- preserve user-entered values across draft patches when possible; if an agent
  patch updates a touched field, keep the user value and show a non-blocking
  "agent updated this field" hint

## Agent guidance / when to use

Use `pi-ask-user`'s skill wording as direct inspiration for the Workspace
`ask_user` guidance. The goal is not to make the agent ask more often; it is to
make the agent stop at consequential decision boundaries instead of guessing.

The injected system prompt or bundled skill for Workspace mode should require a
small decision handshake:

1. **Detect boundary** — classify the next step as `clear`, `ambiguous`,
   `high_stakes`, or `both`.
2. **Gather evidence first** — read code/docs/logs before asking. Do not ask the
   user to decide blind.
3. **Synthesize context** — provide a short neutral summary of current state,
   constraints, trade-offs, and the agent's recommendation when useful.
4. **Ask one focused question** — call `ask_user` for one decision at a time.
5. **Commit decision** — after answer, restate the decision and proceed.
6. **Re-open only on new ambiguity** — do not loop or ask the same trade-off
   again without new evidence.

The agent MUST use `ask_user` before proceeding when the decision is material
and not already determined by user instruction or repo conventions, especially:

- user-visible architecture, schema, API contract, deployment, or security
  changes with meaningful blast radius
- costly-to-undo work: migration, destructive edit, data-loss risk, production
  behavior change, or large refactor whose direction is preference-dependent
- unclear/conflicting requirements where the answer changes implementation
  materially
- multiple valid options where the trade-off is preference-dependent and no
  project convention clearly decides it

The agent SHOULD proceed without asking when repo docs/tests/conventions clearly
imply the choice; state the assumption briefly instead. The agent SHOULD NOT use
`ask_user` for trivial formatting, purely local refactors with identical
behavior, questions answerable by reading files/docs, or trade-offs the user has
already decided explicitly.

Question budget:

- max 1 `ask_user` call per decision boundary in normal cases
- max 2 calls when the first answer is unclear or cancelled
- after 2 unclear/cancelled attempts, stop for high-stakes decisions; for
  ambiguity-only decisions, proceed only if the user explicitly delegates the
  choice ("your call") and state the reversible default chosen

Workspace `ask_user` schema can represent the same simple choice patterns as
`pi-ask-user`, but as generated forms. A single-select decision should map to a
`radio` or `select` field plus optional `textarea` for freeform notes. Independent
multi-choice decisions should map to `multiselect` or a set of checkboxes.

Example generated form for an architecture decision:

```ts
{
  title: "Which caching strategy should we use for v1?",
  context:
    "API p95 latency is high. In-memory cache is simpler but not shared across instances; Redis is scalable but adds infra.",
  schema: {
    wireVersion: 1,
    fields: [
      {
        type: "radio",
        name: "strategy",
        label: "Caching strategy",
        required: true,
        options: [
          {
            value: "in-memory",
            label: "In-memory cache",
            description: "Simpler rollout, weaker horizontal consistency",
          },
          {
            value: "redis",
            label: "Redis cache",
            description: "Better consistency and scalability, more ops overhead",
          },
        ],
      },
      {
        type: "textarea",
        name: "notes",
        label: "Optional constraints or notes",
        required: false,
      },
    ],
  },
}
```

## Principle 3 / pi-tools justification

The repo locks Pi-tool migration behind factory/adapters and requires custom
AgentTools to justify why a generic Pi tool is insufficient. This feature needs
a Workspace-native tool adapter because the product requirement is a Workspace
panel with persisted React form state and answer dispatch through the Questions
browser -> server command channel. Native `pi-ask-user` cannot satisfy those
UI/persistence requirements directly.

Implementation flows through the plugin-owned Pi extension factory
(`createAskUserPiExtensionFactory`) registered by `askUserServerPlugin`. The
`AskUserRuntime` boundary stays narrow and host-side; execution-mode adapters do
not need ask-user-specific sandbox callbacks in v1.

## `pi-ask-user` evaluation

`pi-ask-user` provides a native Pi `ask_user` tool with TUI overlay/inline UI,
selection lists, freeform input, multi-select, timeout, and a useful bundled
skill. It is good prior art only.

Use it as inspiration for agent behavior semantics: prompt guidance,
when-to-ask heuristics, result vocabulary, and cancellation/timeout conventions.
Do not depend on it, wrap it, or use it as the base for Workspace or standalone
implementation.

Do not make it the primary Workspace implementation because it does not provide:

- Workspace panel integration
- generated rich React form rendering
- Workspace persistence semantics
- answer dispatch through the Questions browser -> server command channel
- app-shell primitive components

Recommended stance:

- Share the semantic contract: one `ask_user` tool name, one result vocabulary,
  and compatible cancellation/timeout behavior.
- Workspace mode registers the first-party `ask_user` adapter backed by
  `askUserPlugin`, `AskUserStore`, the existing server -> UI open/focus bridge,
  and the Questions browser -> server command channel.
- Standalone CLI/TUI mode should use a first-party native adapter with similar
  behavior, not the `pi-ask-user` package as-is.
- Never register two adapters under the same `ask_user` tool name in one Pi
  runtime.
- Keep Workspace `askUserPlugin` implementation first-party.

## Acceptance criteria for implementation bead

- Shared schema/types exist and are browser-safe.
- Workspace-owned stable error codes exist for ask-user failures.
- `askUserPlugin` registers a `Questions` panel and optional open command.
- Questions surface resolver opens/focuses the panel for a pending `questionId`.
- Agent/session can create one pending question and block for the answer in
  every supported execution mode: `direct`, `local`, and `vercel-sandbox`.
- Streaming mode can create a draft question, apply validated form patches,
  finalize to `ready`, and only then permit submit.
- Pending question persists across browser reload.
- Abandoned pending questions are handled deterministically.
- Default panel renders text, textarea, select, multiselect, checkbox, radio,
  and number.
- Opening/focusing Questions uses the existing server -> UI bridge; submit and
  cancel use the Questions browser -> server command channel.
- Submitted values resolve the blocking tool call.
- Cancel resolves the tool call with `status: "cancelled"` and
  `reason: "user_cancelled"`.
- Timeout, abort, abandoned, rate-limited, runtime-unavailable, and UI-unavailable
  paths return stable cancelled reasons.
- Primitive form components/hooks are exported for app-shell reuse.
- Unit tests cover schema validation limits, patch validation/finalization,
  stale patch rejection, command payload validation, idempotent submit/cancel,
  transcript event append/list/replay behavior, orphan handling, and
  one-pending-question enforcement.
- Unit tests cover duplicate `patchId` dedup, per-question open-ack timeout,
  rate-limit enforcement, number-field validation, and text pattern safety.
- Integration or harness test covers ask -> render -> submit -> tool result
  through the actual Questions browser -> server command channel.
- Integration test covers two browser tabs racing submits; first wins, second
  renders the accepted answer.
- Integration test covers user typing during streaming patches without losing
  dirty field values.
- Integration test covers accidental pane close while pending, then re-open and
  successful submit.
- Reload-survival test covers create pending -> reload front/session state ->
  Questions panel renders pending question, including reload during draft state
  and re-emitted `questions.opened` ack on remount.
