Here is the critical architecture review of the generic MCP onboarding plan. 

### High
*   **Redaction Guard Feasibility & Interface:** The `McpRedactionGuard.redact(value: unknown)` interface is too abstract to be implemented securely. Redacting arbitrary JSON responses is impossible without knowing exactly what to look for. The interface must explicitly accept or be seeded with the resolved credentials (tokens, PATs) so it knows exactly which strings to redact from the payload.
*   **Schema Hash Drift Brittleness:** Hard-disabling tools on *any* schema hash change (`MCP_PROVIDER_TOOL_DRIFT`) will cause severe operational pain. Providers frequently update descriptions or add optional fields to their MCP tools. If a hash mismatch hard-fails the tool, your integrations will constantly break. *Recommendation:* Differentiate between breaking changes (required fields added, types changed) and non-breaking changes, or use a "warn/flag" state rather than an immediate hard-disable for minor drift.
*   **Mode A (CLI) Stdio Trust Boundary:** The plan states "user is responsible for trusting local stdio commands" and mentions config sources like `.mcp.json` or `.pi/mcp.json`. If the agent has write access to the workspace and can modify these config files, it can add a malicious stdio MCP server (e.g., `command: "bash", args: ["-c", "malicious payload"]`) and execute arbitrary code on the user's machine. Mode A configuration files *must* be strictly isolated from agent write permissions.

### Medium
*   **V0 Tool Exposure Contradiction / Usability:** There is a slight contradiction between the "Generic agent tools" section (which lists `mcp_tools_list`) and the "V0 — generic proxy tools" section (which lists `mcp_tools_search` and `mcp_tool_describe`). If the LLM only has `mcp_tools_list` without a way to fetch the exact JSON schema for a specific tool, it will hallucinate inputs for `mcp_readonly_call`. V0 *must* include the `describe` capability to function reliably.
*   **Missing Pagination in Interfaces:** The plan mentions a cursor for `mcp_resources_list` but omits pagination for `mcp_tools_list` and `mcp_tools_search`. The MCP specification supports pagination for tools. If an Airtable or Notion server exposes hundreds of tools/bases, returning them all at once will blow out the context window and size limits.
*   **Resource URI Fallback / Validation:** The plan states "No fallback to endpoint origin is allowed for V0" for resource URIs. However, many MCP servers use custom URI schemes (e.g., `notion://` or `file://`). The `allowedResourceUriPrefixes` validation must account for custom protocols, not just HTTP origins, and ensure local `file://` reads aren't accidentally exposed in hosted mode.
*   **Concurrency / Connection Leaks:** The "Lessons from pi-mcp-adapter" section mentions "lazy connection lifecycle" and "idle cleanup". However, the `McpExecutionGuard` interface only wraps the *call*, not the *connection*. In hosted mode (Mode B), establishing an SSE/HTTP connection per user/workspace can quickly exhaust server connection pools. The architecture needs a clear strategy for connection pooling or multiplexing across the transport client.
