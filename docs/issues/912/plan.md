# gh-912 minimal live transcript

## Problem

Users need a minimal, live, workspace-native meeting transcript without turning the chat into an audio product or sending local control commands through Pi. Today the agent chat has browser-local slash-command interception (`packages/agent/src/front/slashCommands/registry.ts`, `packages/agent/src/front/chat/session/composerPolicy.ts`) and `PiChatPanel` already accepts host-supplied commands (`packages/agent/src/front/chat/PiChatPanel.tsx`), but it has no live-transcription controller. The workspace has Markdown/file surfaces (`packages/workspace/src/plugins/filesystemPlugin/front/markdown-editor/MarkdownEditorPane.tsx`, `packages/workspace/src/plugins/filesystemPlugin/front/surfaceResolver.ts`) and trusted boot-time server routes, but no statically composed singleton that owns microphone capture, authenticated audio WebSocket proxying, projection, or bounded same-chat review nudges.

The old/archived long-transcript path is not a foundation for this work: the only repository hit is a Pi chat reload regression (`packages/agent/e2e/pi-native-long-transcript-reload.spec.ts`), not a live-audio subsystem. Building on a recorder/upload/chunk workflow would violate the product: audio is stream-only and never retained as an artifact.

The security boundary matters. The browser must not receive Kyutai credentials or supply authoritative workspace/session identities, the transcription backend must stay private and pinned, and transcript text must be treated as untrusted data when a same-chat nudge asks the agent for a recommendation. The current generic boot-time plugin context exposes a raw `workspaceRoot`, and standalone composition uses a local actor; neither is production authority. The current Pi command contract also has no hidden, fail-closed publication operation. Those are feasibility gates, not behavior that implementation may assume already exists.

## Solution

Ship one default-off, app/internal `@hachej/boring-live-transcription` integration, statically composed by the owning application. It is not a runtime-loaded or generally publishable front plugin and does not use `boring.front` provider hot loading. One provider/controller lives above all chat panes, owns capture and the per-browser-tab active session, and injects session-bound command handlers plus the live status strip into each pane. Split panes therefore do not create duplicate controllers or competing microphone owners. The chat session ID captured at confirmed start is immutable for that live session and remains the nudge target even if the visible pane or active session later changes.

The browser contributes exactly one local `/live` command with three forms:

- `/live start "title"`
- `/live stop`
- `/live status`

The command is intercepted before Pi, validates the exact grammar, and returns a compact local notice. Start first shows a compact confirmation containing: **“Audio is streamed to our self-hosted Kyutai service for transcription; this app and service are configured not to retain it.”** Only after acceptance does the authenticated server atomically claim the workspace microphone lease and create `live-transcripts/<UTC timestamp>-<slug>.md`, adding a numeric suffix on collision. The start response creates a `pending-attachment` session, returns one unguessable single-use attachment capability only to that browser call, and starts a 15-second attachment deadline. The browser sends the capability in the first strict binary WebSocket message—not a URL, query, cookie, or logged subprotocol/header—and no audio is accepted before atomic redemption. Redemption binds the exact principal, workspace, originating chat, live session, and browser connection; reuse or mismatch fails with a stable code. If permission, worklet setup, or attachment misses the deadline, the server revision-safely marks the document `interrupted` when possible, releases the lease/runtime UI lock, revokes the capability, and accepts no later attachment. Cancelling before the start request creates no file, requests no microphone permission, and opens no socket.

After the browser receives the start result, it opens the full Markdown document through the existing filesystem surface. The document is read-only in the live UI while that exact path has an active runtime lease; other Markdown documents remain unaffected. This UI lock is advisory UX, not a server write lock. The projector tracks the exact expected file revision (adapter revision/mtime plus content hash) and performs write-if-unchanged through the resolved `Workspace`. If native compare-and-swap is unavailable, the feasibility gate must prove a serialized read/verify/replace seam; any divergence interrupts capture with `live_transcript_revision_conflict`, releases ownership, and does not overwrite or terminally rewrite the externally changed file.

The current chat remains usable. A browser `AudioWorklet` downmixes input to mono, resamples it to 24 kHz float PCM, and emits exact 1,920-sample (80 ms) frames. The browser sends binary PCM frames over an authenticated, same-origin internal WebSocket. Cookie-authenticated HTTP mutations use the host's existing CSRF policy, and every WebSocket upgrade validates a single canonical `Origin` against the configured production allowlist before attachment or lease access; absent, `null`, malformed, multi-valued, or hostile origins fail closed. Bearer deployments still apply the explicit origin policy. The core/full-app composition supplies one narrow host-owned resolver used by both HTTP start/status/stop requests and WebSocket upgrades. It returns `{ principalId, workspaceId, chatSessionAuthority, workspace }` from authenticated request context, workspace membership, and server-owned session ownership. Route code receives the resolved `Workspace`; it never treats headers, query strings, payload workspace/session IDs, or raw roots as authority. Standalone composition may supply a separately named, explicitly local-only resolver for development, but that resolver is not production authentication and cannot satisfy production enablement.

The trusted route verifies the active workspace/session lease and proxies frames to one private, configured Kyutai/Moshi `stt-1b-en_fr` CUDA endpoint. Kyutai URL/token values remain server-only and are never accepted in a query string, browser payload, log, transcript, or client bundle. The server assembles the pinned provider's `Word`, `EndWord`, and `Step` events into timestamped, non-diarized paragraphs and projects the canonical Markdown document no more than once per second, with immediate flushes on a semantic pause, review boundary, and stop/interruption when the expected revision still matches.

Audio bytes are forwarded and discarded; they are never written, replayed, logged, queued durably, or added to the Markdown. A browser-to-server WebSocket close is terminal and immediate: available text is flushed if revision-safe, the session becomes `interrupted`, and there is no browser reconnect, resume capability, or replay, including unload/reload. Only the server-to-Kyutai upstream socket may restart once after a fixed one-second delay, and only while the authenticated browser socket and lease remain open. Audio arriving while the upstream is unavailable is discarded rather than buffered or replayed. The new upstream connection begins a new provider epoch; its dedupe state resets, and provider-relative timestamps receive a monotonic transcript offset derived from the capture clock so projected timestamps never move backward across the gap. A second upstream failure interrupts. Stop flushes and marks the document `complete`; permission denial, browser close, unrecoverable proxy/provider failure, revision conflict, or malformed provider output interrupts. Every terminal path closes capture/sockets, releases the lease, and unlocks the live UI.

A session-bound review scheduler tracks the last reviewed projected revision. At 60 seconds, then every 120 seconds, it coalesces changed text into at most one pending review. It dispatches only when the originating chat is idle, behind admitted user turns; user turns always win and reset/defer pending dispatch. A review boundary first completes a revision-safe projector flush, so the reviewer targets exactly the Markdown revision the user can open.

The review uses one narrowly named, host-defined read-only tool/capability—not “no tools”—which can read only the exact live transcript path at the expected revision through the resolved `Workspace`. No shell, network, write, arbitrary-file, UI, or other tool is available. The hidden instruction treats all returned transcript content as untrusted conversation data and asks for exact `NO_LIVE_RECOMMENDATION` when there is nothing useful.

Publication is fail-closed and atomic. Hidden instruction/input, exact path metadata, tool call, tool result/transcript content, assistant deltas, and the exact sentinel may exist only in bounded server memory for the in-flight review; they are not emitted to browser streams, Boring snapshots/events, logs, native Pi transcript entries, reload reconstruction, or compaction/context persistence. The complete assistant result is buffered until success. Cancellation/error publishes and persists no partial assistant result. Exact `NO_LIVE_RECOMMENDATION` is discarded with no visible/native entry. Any complete non-sentinel result is committed once as one normal assistant message in the originating session, persists in native and Boring history, participates in later same-session model context, and reloads in the same order behind admitted user turns. Near-matches are useful output, not suppressed. This is a hard gate against the pinned Pi API and native session loader (`packages/agent/src/server/harness/pi-coding-agent/sessions.ts`): UI filtering alone is not proof. If same-session context, storage omission, reload/compaction semantics, ordering, or atomic publication cannot all be proven, automated nudges and product implementation stop for owner review; a second agent or visible synthetic user turn is not a fallback.

Implementation shape:

- Add an app/internal package under `plugins/live-transcription/` with shared runtime-neutral schemas/stable errors, a front singleton controller/provider and worklet, server lease/projector/direct Kyutai adapter/review scheduler, tests, package metadata, and operator documentation. Do not advertise or register a dynamic `boring.front` provider and do not add dynamic provider-tree mounting.
- Statically mount exactly one provider above all chat panes at the concrete product-shell composition seam identified by the feasibility gate. Use existing `extraCommands` support in `packages/workspace/src/app/front/WorkspaceAgentFront.tsx` and `packages/agent/src/front/chat/PiChatPanel.tsx`; inject per-pane handlers bound to that pane's session while retaining one controller. Add only the narrow path-read-only and status-strip composition inputs required by this product.
- In `packages/agent/src/front/chat/session/composerPolicy.ts`, add a specifically named pre-admission path limited to exact `/live stop` and `/live status` for the statically injected handler. Do not add reusable `allowWhileBusy` metadata to `SlashCommand`. The always-visible status-strip Stop control remains the authoritative emergency path; unrelated local commands stay blocked while Pi is busy.
- Add a narrowly named server-side live-transcript review operation through the smallest proven seams in `packages/agent/src/core/piChatSessionService.ts`, `packages/agent/src/server/pi-chat/PiAgentSessionAdapter.ts`, and `packages/agent/src/server/harness/pi-coding-agent/createHarness.ts`. Change browser/shared command schemas only if the feasibility proof demonstrates they are unavoidable; do not expose a generic hidden-turn request. Test native persistence/reload through `packages/agent/src/server/harness/pi-coding-agent/sessions.ts` and its loader tests.
- Define an internal resolver contract owned by core/full-app composition, with `packages/core/src/app/server/createCoreWorkspaceAgentServer.ts` and `packages/core/src/app/server/coreWorkspaceBridge.ts` as the production auth/membership/session-ownership precedents. Use the same resolver for HTTP and upgrade paths and inject it into a narrowly named live-transcription route factory. Do not add auth or `Workspace` to the generic external-plugin API. `packages/workspace/src/app/server/createWorkspaceAgentServer.ts` may wire only an explicitly local-only development implementation.
- Reuse resolved `Workspace` operations rather than raw roots. Path containment/collision handling stays in the adapter, per `docs/kanzen/procedures/coding-invariants.md`. Prove native CAS or the serialized expected-revision fallback before capture/projector implementation.

## Decisions

1. **Exactly three command forms under one browser-local `/live` command.** Rationale: composer interception guarantees control text never reaches Pi and avoids a second command transport. Unknown subcommands, missing/extra arguments, unquoted/empty titles, `/live pause`, and a second start return compact usage/status notices with no side effects. Only the statically injected `/live stop` and `/live status` handlers receive a narrowly named busy pre-admission exception; every other local command remains blocked while Pi is busy. An always-visible live status strip beside each chat pane exposes only status and Stop, so microphone shutdown never depends on Pi becoming idle.
2. **Confirmation precedes every persistent, media, or network side effect.** Rationale: consent is meaningful only if cancel leaves no transcript file, microphone lease, `getUserMedia` request, or WebSocket. The disclosure accurately states the controlled app/service configuration rather than claiming proof about arbitrary infrastructure.
3. **One microphone owner/live session per workspace in one owning app process.** Rationale: browser-only locks fail across tabs, while a process-local lease cannot guarantee uniqueness across replicas. Enablement requires one owning app process/replica for the workspace; no distributed lease platform is added. The in-memory lease is keyed by workspace and associated with authenticated principal, browser connection, immutable originating chat session, transcript path, and state. A competing start receives stable `live_transcript_already_active` status and never steals ownership.
4. **Stream-only audio.** Rationale: privacy and scope. Name and bound every app-owned audio-bearing location: the worklet carries at most one 1,920-sample frame plus bounded resampler carry; the browser socket owns at most one unsent frame; the proxy owns at most one decoded frame and one in-flight upstream send; Moshi owns only its documented bounded streaming state. Overflow interrupts instead of queueing. During the one upstream restart delay, incoming audio is discarded, not retained or replayed. Terminal cleanup zeroes/removes local frame references, closes both sockets, and asserts all app-owned audio queue byte counters are zero. No MediaRecorder, upload, object storage, temp file, database blob, playback, raw chunks, crash payload, request-body log, or recovery buffer is introduced.
5. **Browser normalizes audio to Kyutai's fixed input.** Rationale: an `AudioWorklet` keeps capture off the main thread and makes the wire format deterministic: mono float32 little-endian, 24,000 samples/second, exactly 1,920 samples/80 ms per binary message. A final partial frame is discarded on stop; text events are flushed.
6. **Core/full-app-owned authenticated resolver, CSRF/origin policy, and pending attachment capability; private pinned upstream.** Rationale: HTTP and WebSocket upgrade must converge on one authenticated principal/workspace/session authority and resolved `Workspace`, but ambient cookie authentication does not prevent cross-site requests and same-principal tabs must not race to attach. HTTP mutations reuse the host CSRF policy; upgrades validate the configured origin allowlist. Confirmed start returns one 256-bit single-use capability bound to scope/session/connection with a 15-second deadline, sent only as the first strict binary message and stored server-side only as a hash. Browser credentials, presented IDs, and the capability alone are not authority. The internal proxy alone reads server environment secrets. Production configuration pins the Kyutai/Moshi server image and model revision for `stt-1b-en_fr` and verifies CUDA readiness before enablement.
7. **Browser close is terminal; one upstream-only restart.** Rationale: a browser reconnect would require resumable capability, introduce an audio gap/replay security surface, and contradict a connection-bound lease. Any browser-to-server close interrupts immediately. Only the server-to-Kyutai socket retries once after one second while the authenticated browser socket remains open. There is no replay or exponential loop. The retry starts a new provider epoch, resets provider-event dedupe, and applies a monotonic capture-clock offset to provider timestamps.
8. **Canonical Markdown is the sole transcript artifact and live UI.** Rationale: `live-transcripts/<timestamp>-<slug>.md` is inspectable by user and restricted reviewer and reuses the existing Markdown surface. It contains title, start time, state, and timestamped paragraphs; no diarization. The exact path is UI-read-only only while its runtime lease is active, then unlocks. There are no transcript cards or parallel transcript artifacts.
9. **Revision-safe projection at most once per second, plus semantic flushes.** Rationale: cap filesystem/SSE/editor churn without clobbering external writers. Every whole-document replace checks the projector's expected adapter revision/mtime and content hash. Native CAS is preferred; otherwise a proven serialized verify-before-replace seam is required. Divergence stops without overwriting. `Word` accumulates text, `EndWord` finalizes token boundaries, and `Step` advances provider time/pause detection; dedupe is scoped to one provider epoch.
10. **Same-chat nudges are hidden, changed-only, deferred, and coalesced.** Rationale: the originating conversation supplies useful context but cannot be disrupted. First eligibility is 60 seconds after start; subsequent eligibility is 120 seconds after the last dispatched review. No projected revision change means no turn. Busy/user-pending state defers one coalesced review until idle, behind user work. Changing the visible pane does not retarget it.
11. **Transcript is untrusted; the one read tool and publication are restricted.** Rationale: spoken content can contain prompt injection. The server supplies the instruction; the host-defined tool reads only the exact expected-revision Markdown through `Workspace` and no other tool exists. Hidden material and sentinel never enter durable/native/browser history. Only a complete useful assistant result is atomically published as a normal assistant message. Exact `NO_LIVE_RECOMMENDATION` alone is suppressed; prefixes, suffixes, or whitespace-modified text are not.
12. **Graceful cleanup is guaranteed; hard-process-loss residue is documented.** Rationale: Fastify/process graceful-close hooks use a bounded shutdown deadline to stop capture, flush revision-safe text as `interrupted`, close sockets, and release leases on flag disable/restart. A hard process/container kill cannot run that hook: its in-memory lock disappears and the file is editable after restart, but Markdown metadata may still say `active`. MVP accepts and documents that stale metadata; it does not claim crash recovery or silently add startup reconciliation.
13. **No integration with speculative recording infrastructure.** Rationale: no pause, reload recovery, retained audio, playback, chunks, diarization, transcript cards, modes, multiple simultaneous sessions, provider abstraction, generic media bus, generic scheduler, command capability, or background-turn platform is required.
14. **Stable errors and architectural boundaries remain mandatory.** Rationale: new shared errors are added to the owning canonical enum and schemas use runtime-neutral types (`Uint8Array`, never `Buffer` or `node:*` under `src/shared/**`). Routes receive a resolved `Workspace`, not a root path; workspace base front/shared code does not value-import the agent.

## Flag / Abstraction
- Needed?: Yes, server-owned `BORING_LIVE_TRANSCRIPTS_ENABLED`, default off. The `/live` contribution, status strip, HTTP endpoints, and proxy upgrade are absent/rejected when off; existing chat/files behavior is unchanged. Flag-on startup fails closed unless the pinned Kyutai configuration and single-owning-process deployment prerequisite are attested.
- Path: One app/internal, statically composed `@hachej/boring-live-transcription` package with direct Kyutai/Moshi `stt-1b-en_fr` mapping. Inject only a core/full-app-owned authority resolver, existing chat/file surface capabilities, one exact-path review tool, and server-only Kyutai configuration. Do not define a dynamic front plugin, generic STT provider, recorder, audio store, distributed lease, review job, hidden-turn API, or command platform.
- Rollback: Gracefully disable `BORING_LIVE_TRANSCRIPTS_ENABLED` and restart the single owning process. Graceful close interrupts revision-safe active documents, closes sockets, releases leases, and leaves Markdown readable/editable. A hard kill releases the runtime lock on restart but may leave file metadata `active`; this documented residue is not recovery. No schema/data migration or audio cleanup is required. Revert the static package/app composition separately if code rollback is needed.

## Out of Scope

- Pause/resume.
- Browser reconnect, reload recovery, resumable capabilities, audio replay, or crash recovery; browser close/reload interrupts.
- Distributed leases or multi-replica ownership while enabled; one owning process/replica is a prerequisite.
- Retained audio of any kind, recordings, uploads, playback, downloadable audio, temporary audio files, database/object-store audio, or durable retry queues.
- Audio chunks as visible or persisted artifacts.
- Diarization, speaker identification, transcript cards, summaries as a separate artifact, editing in the live UI while active, modes, language/model/provider selectors, or multiple live sessions per workspace.
- Additional commands such as restart, rename, export, retry, configure, devices, pause, or resume.
- Generic STT/provider abstractions, WebSocket/audio buses, job schedulers, hidden-agent/background-turn platforms, reusable busy-command capabilities, dynamic provider mounting, or integration with an old recording subsystem.
- Query-string credentials, browser-visible Kyutai secrets, direct browser-to-Kyutai access, public Kyutai exposure, local-only auth presented as production auth, or unpinned deployments.
- A recommendation when content has not changed, visible nudge prompts, retargeting to the currently visible chat, parallel review sessions, transcript-driven unrestricted tools, or suppressing anything other than exact `NO_LIVE_RECOMMENDATION`.
- Proving absence of audio from arbitrary process heap, host memory, infrastructure packet capture, or operator storage outside the attested app/service configuration and named controlled locations.
- Startup reconciliation of stale `active` Markdown after a hard process kill; the documented MVP behavior is unlocked/editable content with possibly stale metadata.
