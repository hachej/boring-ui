## (b2) HARD INVARIANT #2 — single-writer-per-conversation (scopes ALL pump/coordinator reasoning)
**Exactly one bridge process owns a given conversationKey at a time.** BCH-004's pump ownership (`activePumps`, a process-local Map spike:28) and BCH-001's DrainCoordinator (process-local per-session mutex/phase) are correct ONLY under this invariant. The Chat SDK runs handlers under `concurrency:'concurrent'` with NO cross-process lock (chat d.ts:572), so two replicas handling one conversation would double-render and double-drain.
- **Deployment guarantee (this pass):** the host routes all events for a conversationKey to a single bridge process — via sticky dispatch (one bridge process per workspace, or per Slack team/enterprise id), or a consistent-hash router keyed on conversationKey in front of N bridge processes. BCH-005's README documents the concrete guarantee.
- **Durable vs process-local (do not conflate):** the SESSION STORE (BCH-003, `conversationKey → {sessionId,startIndex}`) is DURABLE and cross-instance-correct via an OWNER-TOKEN CAS state machine (proven by two-store-instances-over-one-backing + delayed-allocation-past-TTL tests). The PUMP (BCH-004) and the DrainCoordinator (BCH-001) are PROCESS-LOCAL and correct only under single-writer. A create-race is resolved DURABLY by the store's CAS; a pump/drain race is resolved only WITHIN the single owning process.
- **Deferred / Follow-up (horizontal scale — NOT this pass):** to run multiple bridge replicas per conversation, add a DURABLE pump lease keyed by conversation/session (owner token + TTL, on the same backing as BCH-003 — the CAS primitive already exists) so only the lease-holder posts while non-holders enqueue via the durable store and drop their local pump. Separate bead; scope when multi-replica-per-conversation is actually needed.
