Boring UI auth plan

Google Signup for Any Child App

Status: review ready · Scope: @hachej/boring-core + child-app docs/skill · Source: docs/plans/google-signup-child-app-plan.md

Goal: make Google signup/signin a near-zero-code opt-in for child apps using the existing core auth stack.

What this is

  • better-auth already powers core auth
  • Google is one better-auth social provider
  • core remains the auth owner

What ships

  • Google config in core
  • features.google_oauth = true
  • socialProviders.google
  • GoogleAuthButton
  • Google callback route

What stays small

  • no second auth system
  • no GOOGLE_OAUTH env flag
  • no generic provider UI
  • no invite OAuth state carry

Core idea

@hachej/boring-core already owns auth.

Google should be added as one social provider inside the existing better-auth provider, not as a second provider system.

  • no separate auth architecture
  • no custom token/session model
  • no app-specific Google wiring in each child app
  • no forked auth pages required for normal usage

Goal

A stock child app should only need:

  • GOOGLE_CLIENT_ID
  • GOOGLE_CLIENT_SECRET
  • features.google_oauth = true

Then core should provide Google on stock sign-in, Google on normal stock sign-up, callback handling through the existing auth stack, and one reusable GoogleAuthButton for branded child apps.

Why this matters

  • child apps can enable Google with a tiny config delta
  • core remains the canonical auth owner
  • branded child apps reuse one small primitive instead of re-learning better-auth
  • we avoid auth drift across child apps

User workflows

Workflow A — stock child app

  1. operator sets GOOGLE_CLIENT_ID and GOOGLE_CLIENT_SECRET
  2. operator sets features.google_oauth = true
  3. user visits /auth/signin
  4. user sees Continue with Google
  5. user signs in and lands back in the app authenticated

Workflow B — stock signup page

  1. user visits /auth/signup
  2. user sees Google plus the existing email form
  3. user can choose Google or email signup
  4. existing email signup behavior stays unchanged

Workflow C — invite signup

  1. invited user lands on /auth/signup?invite_token=...
  2. user sees the existing email signup flow only
  3. user completes the existing invite-aware email path
First-pass simplification: choose correctness over OAuth state-carry complexity.

Workflow D — branded child app

  1. child app overrides core auth pages
  2. child app imports GoogleAuthButton from core
  3. child app keeps its own branding/layout while reusing core OAuth behavior

Non-goals for this first pass

no generic social-provider registry no second auth provider abstraction no invite-token carry across OAuth redirects no Google One Tap no provider-specific landing-page config no auth override API redesign

Locked decisions in this draft

These are the simplifications this plan asks the reviewer to approve.

Provider model

Google comes through better-auth inside the existing core auth stack. No separate auth system.

Config shape

No GOOGLE_OAUTH env flag. One app-level TOML flag plus credentials.

Invite flow

Invite signup stays email-only. No OAuth invite-token carry in v1.

Front-end seam

Ship GoogleAuthButton, not a generic provider UI.

Routing scope

Add an explicit Google callback route. No callback-routing refactor.

Failure mode

Missing credentials disable Google cleanly. App still boots.

Proposed implementation

1. Config surface

  • env: GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET
  • TOML: features.google_oauth = true
  • types: add auth.google, features.googleOauth, runtime/capability fields
Rule: Google resolves on only when the TOML flag is true and both credentials exist.

2. better-auth wiring

socialProviders: {
  ...(config.auth.github ? { github: {...} } : {}),
  ...(config.auth.google ? { google: {...} } : {}),
}

Do not add provider-specific custom logic beyond the minimum better-auth config needed for Google.

3. Front-end auth UX

  • ship one helper: GoogleAuthButton
  • visibility driven by useConfig()
  • internally calls authClient.signIn.social({ provider: 'google', callbackURL: '/' })
  • optional callbackURL prop is acceptable; default stays '/'
  • show on stock sign-in and normal stock sign-up
  • hide on sign-up pages with invite_token

4. Callback-route audit

  • audit route map / front utils
  • audit CoreFront route registration
  • audit server-side auth-page allowlist/public-page handling
  • audit the SPA-shell auth-page allowlist in createCoreWorkspaceAgentServer.ts
  • add route tests for /auth/callback/google
Important: core's better-auth base path is /auth, not /api/auth. So /auth/callback/google is a real auth callback path in this app.
Scope rule: add the explicit Google callback path everywhere needed. Do not generalize callback routing yet.

5. Child-app override seam

Export GoogleAuthButton from @hachej/boring-core/front. Do not ship a generic provider UI in v1.

6. Docs + skill

  • update core docs with Google signup env + redirect URI setup
  • update child-app setup skill
  • document callback URLs:
local: http://localhost:3000/auth/callback/google
prod:  https://<app-domain>/auth/callback/google

What exactly must be implemented

  1. Core config — add GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET, features.google_oauth = true, and teach TomlAppConfig.features to read google_oauth.
  2. better-auth wiring — pass socialProviders.google into the existing core auth setup.
  3. Capabilities/runtime flags — expose features.googleOauth and core.auth.google, and update buildRuntimeConfigPayload() so the frontend actually receives the flag.
  4. Reusable front-end primitive — add GoogleAuthButton that calls authClient.signIn.social({ provider: 'google', callbackURL: '/' }).
  5. Stock auth pages — use it on stock sign-in and normal stock sign-up; hide it for invite-token signup.
  6. Callback route — add /auth/callback/google, treat it like a public auth callback route, and update the SPA-shell auth-page allowlist in createCoreWorkspaceAgentServer.ts.
  7. Docs + skill — update CORE.md and the child-app setup skill; document BETTER_AUTH_URL, CORS_ORIGINS, and callback URLs.
  8. Tests — config, auth wiring, callback route, visible/hidden button behavior, and existing email auth still passing.

Implementation task groups

Verify: config tests cover enabled + disabled cases.

Update CORE.md and setup skill; include BETTER_AUTH_URL + CORS_ORIGINS callouts.

Acceptance criteria

Product

  • child app can enable Google signup without forking core auth pages
  • default sign-in shows Google when enabled
  • default sign-up shows Google for normal signup flow
  • invite-token sign-up stays email-only
  • Google is absent when feature/credentials are missing
  • custom child-app auth pages can import GoogleAuthButton

Technical

  • config validation covers Google env + flag resolution
  • better-auth receives Google provider config only when valid
  • /api/v1/config exposes features.googleOauth
  • /api/v1/capabilities exposes Google auth availability
  • core.auth.google means usable now, not merely configured
  • /auth/callback/google is mounted and public as an auth callback route
  • existing GitHub behavior keeps working

Tests

  1. loadConfig resolves googleOauth when TOML flag + creds are present
  2. loadConfig keeps googleOauth=false when creds are incomplete
  3. createAuth includes Google provider when configured
  4. CoreFront mounts /auth/callback/google
  5. SignInPage renders Google only when enabled
  6. SignUpPage renders Google only for normal signup flow
  7. SignUpPage hides Google when invite_token is present
  8. clicking Google calls signIn.social({ provider: 'google' ... })
  9. existing email sign-in/sign-up tests still pass

Risks / watchouts

  • Docs drift: docs still describe social auth as deferred / GitHub-specific in several places
  • Callback mismatch: Google will fail if BETTER_AUTH_URL is wrong
  • Invite flow complexity: do not invent OAuth state carry in this first pass
  • Capability ambiguity: define core.auth.google as usable now
  • Over-design guardrail: do not build a generic provider framework yet

Review questions

  1. Provider model: approve treating Google as one better-auth social provider inside the existing core auth stack?
  2. Config shape: approve TOML flag + credentials, with no separate GOOGLE_OAUTH env flag?
  3. Invite behavior: approve hiding Google on invite-token signup instead of solving OAuth state carry now?
  4. Front-end seam: approve shipping only GoogleAuthButton in v1?
  5. Routing scope: approve adding an explicit Google callback path instead of generalizing callback routing now?
  6. Redirect default: approve defaulting the Google button callback to '/' for the first pass?

If the answer to all six is yes, this plan is ready to implement.

Nice-to-have later, not in this first pass

  • generic socialProviders runtime type
  • GitHub + Google unified provider list
  • Google One Tap
  • provider icons/theme slots
  • provider-specific callback / landing route config
