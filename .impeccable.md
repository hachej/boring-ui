# Impeccable design context — boring-ui-v2

## Design Context

### Users
Developers embedding a coding-agent pane in their own apps, and end-users chatting with an agent inside an IDE/workspace shell. Primary context: a focused coding session — the chat sits beside an editor and file tree. Users read code, tool outputs, and reasoning passages for minutes at a time. Reading comfort on long bot messages and surgical composer speed are the two most-used capabilities.

### Brand Personality
Three words: **precise, calm, editorial**.
- **Precise** — everything on screen is intentional. No stray pixels, no speculative decoration, no "filler" UI. The grid is tight; radii, sizes and weights form a visible ramp.
- **Calm** — the panel recedes so the conversation can be the subject. Dark surfaces, low-chroma neutrals, one accent used sparingly.
- **Editorial** — message prose feels like it was set by a designer, not dumped into a div. Line-length caps, leading, heading rhythm, inline-code chips, and code-block typography all read like a magazine column over a log dump.

Reference: https://vercel.com/templates/next.js/chatbot — this is the canonical target for "AI chat done well". Vercel's `ai-elements` registry (Tool, Artifact, Attachments, CodeBlock, Message, PromptInput, Reasoning) is the primitive vocabulary; we adopt it 1:1 and style *within* the primitives' contracts, not around them.

Anti-references: the generic "AI chatbot" look — glowing cyan/purple gradients, glassy cards, border-left colored stripes, icon-above-every-heading template, bouncy elastic animation, bubble + huge drop shadow.

### Aesthetic Direction
**Refined minimal, dark-first.** Card-shaped surfaces only when they carry meaning (tool calls, artifacts, attachments); prose and bare responses sit on the page.
- Dark surface ramp: `background`, `card`, `popover` all within ~0.02 lightness of one another — separation via borders and opacity, not big elevation jumps.
- One accent (the `primary` OKLCH token). Used for: the send button, the user-message tint, interactive focus rings. Never for decoration.
- Radius ramp: `sm → 2xl` at 6/8/10/12/16/24px. Bubbles, tool cards, attachment pills use `lg` (10px). Inline chips use `sm`.
- Light theme ships but is derivative — we prove design in dark first.
- No CSS we're banned from writing: no gradient text, no colored `border-left > 1px` accent stripes, no glassmorphism blur for chrome.

### Design Principles
1. **The conversation is the interface.** Bubble chrome, buttons, and composer never compete with the message surface. If an affordance isn't needed right now, it disappears (truly — it takes zero layout room, not just zero opacity).
2. **Symmetry before tint.** User and assistant share padding, radius, typography, line-height — the only honest difference is a subtle fill tint. Asymmetric paddings = perception of uneven weight and the chat "leaning".
3. **Tokenized, never ad-hoc.** Radii, spacing, font-sizes, and line-heights come from a visible ramp in `globals.css`. If a value isn't in the ramp, it's wrong or the ramp is missing a step.
4. **Reveal reserves nothing.** Hover affordances (action bars, trailing chrome) animate from `display: none` into place — they do NOT sit invisible in the layout. Idle messages must be tight.
5. **Editorial prose inside bot messages.** Prose has real typography (paragraph rhythm, list markers, cap-height code chips, heading-as-weight-not-size). Code blocks own their own surface — no double-wrapping them in another card.
6. **One moment of motion per state.** Message arrival, tool collapse/expand, composer focus — each gets a single 120–180ms ease-out tween. No layered animations.
7. **Every pixel earns its place.** Remove, then verify nothing was lost. The Vercel-chatbot aesthetic is ~80% what you *didn't* put on screen.

### Theme
Dark is the primary theme (developer tool, long focused sessions, IDE-adjacent). Light ships as a derivative mapping of the same tokens and passes contrast, but design is signed off in dark.

### Accessibility
- Contrast floor: WCAG AA for body prose, AA-large for disabled/placeholder.
- Every interactive surface has a visible focus ring (`ring-2 ring-ring/40`).
- Hover-revealed affordances must also appear on `:focus-within` (keyboard users).
- `prefers-reduced-motion` disables non-essential transitions.
