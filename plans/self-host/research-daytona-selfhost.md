# Research: Daytona open-source/self-hosting for boring-ui sandbox backend

## Summary
Daytona is a credible self-host candidate for a boring-ui agent sandbox/workspace backend if we are comfortable operating a Docker/Sysbox-based compute plane and accepting AGPL-3.0 obligations. It gives us a ready control plane, SDK/API, preview proxy, SSH gateway, snapshots, volumes, lifecycle policies, and per-sandbox process/filesystem APIs, but it is not a zero-burden library: self-host means running PostgreSQL, Redis, OIDC, registry, S3-compatible storage, proxy, runner(s), TLS/DNS, and host-level container runtime plumbing.

Recommendation: use Daytona as a **pilot backend behind boring-ui's Workspace/Sandbox adapter**, not as the only self-host plan yet. Start with one Linux VM/bare-metal runner using the official Docker Compose/domain deployment, validate untrusted-agent isolation and persistence semantics, and defer Kubernetes/containerd deployment until the runner/runtime story is proven.

## Findings
1. **Architecture fits boring-ui's backend boundary: interface/control/compute planes plus sandbox tool APIs.** Daytona exposes SDKs, CLI, dashboard, MCP, SSH; a NestJS REST API handles auth, sandbox lifecycle, snapshots, volumes, and resource allocation; proxy routes `{port}-{sandboxId}.{proxy-domain}` traffic; runners execute lifecycle jobs; an in-sandbox daemon exposes Toolbox APIs for files, Git, process execution, PTY/logs, etc. This maps well to a boring-ui runtime adapter where boring-ui calls Daytona for sandbox lifecycle and command/file ops rather than managing containers directly. [Architecture](https://www.daytona.io/docs/en/architecture/) [Sandboxes](https://www.daytona.io/docs/en/sandboxes/)

2. **Self-host install is Docker Compose first, not a small embedded component.** OSS deployment uses `docker compose -f docker/docker-compose.yaml up -d` and includes API, proxy, runner, SSH gateway, PostgreSQL, Redis, Dex OIDC, registry, MinIO, MailDev, Jaeger, PgAdmin. Domain deployment needs a public domain, DNS records including wildcard proxy DNS, Caddy with DNS-provider module for wildcard TLS, ports 80/443/2222, and generated secrets replacing placeholders. Minimum host is documented as 4 GB RAM, 8 GB recommended. [OSS deployment](https://www.daytona.io/docs/en/oss-deployment/)

3. **Runtime requirement is effectively Docker + Sysbox for runners; no KVM is required for the default container path.** Runner install docs state supported architecture is AMD64/x86_64, Docker is installed if absent, systemd is required, and `CONTAINER_RUNTIME` defaults to `sysbox-runc`. Sysbox is a specialized OCI runtime, not a VM/KVM runtime; it improves container isolation with user namespaces and VM-like system containers, but explicitly does not provide the same isolation level as VM-based runtimes. [Runners](https://www.daytona.io/docs/en/runners/) [Sysbox README](https://github.com/nestybox/sysbox?tab=readme-ov-file)

4. **Kubernetes exists, but looks operationally risky today for boring-ui self-host v1.** Daytona docs mention Helm charts for custom regions on Kubernetes, but a recent helm-charts issue reports the runner DaemonSet attempting host-level Docker + Sysbox + XFS quota installation via `nsenter/systemctl`, failing on containerd-only clusters and looping forever. Treat Kubernetes/containerd as a later track, not the first self-host target. [Runners](https://www.daytona.io/docs/en/runners/) [Helm chart issue](https://github.com/daytonaio/helm-charts/issues/11)

5. **Persistence model is strong but must be matched carefully to boring-ui workspace expectations.** Stopped sandboxes keep filesystem state but lose memory; archived sandboxes move filesystem state to object storage and can later restart; pause/memory persistence is documented for VM-based runners; volumes are S3-backed persistent read-write directories shareable across sandboxes; snapshots are reusable OCI/Docker-based templates; sandbox snapshots can capture filesystem and, for Windows, optionally memory. Boring-ui should model Daytona sandboxes as persistent workspaces only if stop/archive/delete TTLs are controlled explicitly. [Sandboxes](https://www.daytona.io/docs/en/sandboxes/) [Snapshots](https://www.daytona.io/docs/en/snapshots/) [Architecture](https://www.daytona.io/docs/en/architecture/)

6. **API/SDK integration shape is favorable.** TypeScript SDK is `@daytona/sdk`; `new Daytona({ apiKey, apiUrl, target })` or env vars `DAYTONA_API_KEY`, `DAYTONA_API_URL`, `DAYTONA_TARGET`; create a sandbox, then call process/file APIs (`sandbox.process.executeCommand(...)`, uploads/downloads, etc.). REST OpenAPI specs are linked for platform and Toolbox APIs. This is enough for a `DaytonaWorkspace` adapter with lifecycle, exec, file, preview URL, labels, and TTL methods. [TypeScript SDK](https://www.daytona.io/docs/en/typescript-sdk/) [Sandboxes](https://www.daytona.io/docs/en/sandboxes/)

7. **Security isolation is better than plain Docker but not VM-grade by default.** Daytona says each sandbox has isolated process/network/filesystem/IPC namespaces, dedicated vCPU/RAM/disk, per-sandbox firewall rules, and optional egress restriction. Compose defaults `INTER_SANDBOX_NETWORK_ENABLED=false`, disabling inter-sandbox communication. However, Sysbox itself notes it is not equivalent to VM isolation; for malicious multi-tenant untrusted code, boring-ui should classify this as container-hardening isolation, not a hard security boundary. [Architecture](https://www.daytona.io/docs/en/architecture/) [Sandboxes](https://www.daytona.io/docs/en/sandboxes/) [OSS deployment](https://www.daytona.io/docs/en/oss-deployment/) [Sysbox README](https://github.com/nestybox/sysbox?tab=readme-ov-file)

8. **Operational burden is moderate-to-high.** We would own Docker/Sysbox host health, runner capacity, S3/MinIO storage durability, registry garbage collection, Postgres backups/migrations, Redis, Dex/OIDC config, Caddy/TLS/DNS, firewalling, observability, quota/rate-limit configuration, snapshot activation, and secure handling of API/runner/proxy/SSH gateway keys. Official docs say self-hosted deployments are community-supported via GitHub/Slack unless dedicated support is arranged. [OSS deployment](https://www.daytona.io/docs/en/oss-deployment/) [Runners](https://www.daytona.io/docs/en/runners/)

9. **Costs: software can be free to self-host, but infra is ours; managed prices are useful benchmark only.** Daytona OSS is AGPL-3.0 and can be deployed anywhere; managed cloud pricing lists compute at $0.0504/vCPU-hour, memory at $0.0162/GiB-hour, storage at $0.000108/GiB-hour after 5 GB free, H100 at $3.95/hour, RTX PRO 6000 at $3.03/hour, and $200 free compute. For boring-ui self-host, expect VM/bare-metal cost plus storage/egress/backup/support costs instead of Daytona usage bills. [OSS deployment](https://www.daytona.io/docs/en/oss-deployment/) [Pricing](https://www.daytona.io/pricing)

10. **Licensing is a strategic constraint.** Daytona repository and OSS docs identify AGPL-3.0. Running unmodified Daytona internally is likely fine, but modifying Daytona and offering it over a network can trigger AGPL source-sharing obligations for those modifications. Keep boring-ui integration as an external adapter using SDK/API where possible; avoid vendoring/modifying Daytona unless legal approves. [GitHub repository](https://github.com/daytonaio/daytona) [OSS deployment](https://www.daytona.io/docs/en/oss-deployment/)

## Boring-ui self-host recommendation
- **Plan A pilot:** run Daytona OSS on one dedicated Linux AMD64 host/VM via Docker Compose domain deployment. Use generated secrets, DNS-only wildcard proxy records, Caddy TLS, closed auxiliary ports, `INTER_SANDBOX_NETWORK_ENABLED=false`, and explicit egress policy.
- **Adapter scope:** implement a `DaytonaSandbox/Workspace` backend that creates sandboxes from a boring-ui base snapshot, executes commands through Toolbox/SDK, maps file operations, exposes preview URLs, sets labels for ownership/beads/tasks, and controls `auto_stop_interval`, `auto_archive_interval`, and `auto_delete_interval`.
- **Persistence policy:** for long-lived boring-ui workspaces, disable surprise deletion, use volumes or archived sandboxes for durable state, and test stop/start/archive/restore before claiming workspace persistence.
- **Security stance:** acceptable for trusted/single-tenant or semi-trusted self-host agents. For arbitrary hostile user code, require stronger isolation review or VM/Kata/Firecracker alternative.
- **Do not start with Kubernetes:** use bare VM/bare metal first. Revisit Helm only after Daytona runner supports our cluster runtime cleanly or we accept host mutation by DaemonSet.
- **Decision gate:** adopt Daytona only if a 1-2 week spike proves: install reproducibility, sandbox boot latency, command/file API coverage, snapshot build flow, storage durability, network egress controls, crash recovery, and acceptable host escape risk.

## Gaps
- Did not run Daytona locally; install friction, actual resource use, and failure recovery need hands-on validation.
- Could not confirm current production-grade Kubernetes/containerd support beyond docs and one issue; needs a cluster spike if Kubernetes is required.
- Exact host hardening requirements for malicious-code multi-tenancy need security review beyond vendor docs.
- Need legal review of AGPL impact on distributing a boring-ui + Daytona self-host bundle.
