# Research: self-hosted sandbox options for boring-ui

## Summary
For normal commodity VPS, the safest practical path is **Docker/OCI containers hardened with gVisor (`runsc`)**, plus strict egress/filesystem policy; it does **not** require `/dev/kvm` and works with Docker/Kubernetes on Linux. For stronger client-safe isolation, move staged production to **microVMs on bare metal or nested-virt-capable VMs**: Firecracker directly, E2B self-host, or BoxLite all require `/dev/kvm`, which many VPS plans do not expose. Daytona OSS is the fastest full product to stand up on a VPS via Docker Compose, but its OSS compose path is a larger app stack and appears container/Docker-runner based, not a KVM microVM boundary.

## Findings
1. **Normal VPS constraint is decisive: `/dev/kvm` is the split point.** Firecracker requires KVM and read/write `/dev/kvm`; Firecracker docs explicitly recommend EC2 `.metal` because EC2 only supports KVM there. E2B and BoxLite inherit the same constraint. [Firecracker](https://github.com/firecracker-microvm/firecracker/blob/main/docs/getting-started.md), [E2B](https://github.com/e2b-dev/infra/blob/main/DEV-LOCAL.md), [BoxLite](https://docs.boxlite.ai/getting-started)
2. **Daytona OSS is the easiest full self-host product on a VPS.** Its official OSS guide is Docker Compose on Linux cloud/bare-metal or macOS, with API/proxy/runner/SSH gateway plus data services; it does not document KVM as a default requirement. [Source](https://www.daytona.io/docs/en/oss-deployment/)
3. **gVisor is the best VPS-compatible isolation upgrade.** It provides an OCI runtime (`runsc`) for Docker/Kubernetes, with a userspace application kernel intercepting syscalls instead of passing app syscalls directly to the host kernel. [Install](https://gvisor.dev/docs/user_guide/install/), [Model](https://gvisor.dev/docs/)
4. **bwrap/nsjail are useful but policy-heavy.** Both rely on Linux namespaces/seccomp/cgroups and share the host kernel; bubblewrap explicitly says it is not a complete ready-made sandbox and protection depends on chosen arguments. [bwrap](https://github.com/containers/bubblewrap/blob/main/README.md), [nsjail](https://github.com/google/nsjail/blob/master/README.md)
5. **Direct Firecracker gives the strongest DIY boundary but highest ops cost.** You must manage guest kernels/rootfs, TAP devices/NAT, jailer, cgroups and lifecycle; production use is more platform engineering than app integration. [Source](https://github.com/firecracker-microvm/firecracker/blob/main/docs/getting-started.md)

## Recommended staged path
1. **Stage 0: lock down the boring-ui contract.** Treat all agent code as hostile: no host Docker socket, no host project mounts except explicit workspace copies, per-client/per-run identity, hard CPU/RAM/PID/disk/time quotas, default-deny egress with allowlist/proxy, audit logs, teardown/reap jobs.
2. **Stage 1: client-safe VPS launch.** Run ephemeral OCI containers under **gVisor `runsc`**; optionally wrap commands with nsjail/bwrap inside the container for per-process limits. This works on normal VPS and keeps ops small.
3. **Stage 1b: evaluate Daytona OSS if you want a turnkey sandbox service.** Deploy on a separate VPS, replace defaults/secrets, disable inter-sandbox networking, verify cgroup limits and whether runner can use gVisor/containerd in your setup.
4. **Stage 2: high-risk clients / stronger isolation.** Move runners to bare metal or a provider that exposes nested KVM; pilot **BoxLite** for persistent workspaces or **E2B** if SDK/API compatibility is valuable.
5. **Stage 3: hardened platform.** Standardize on Firecracker-based microVMs, either via E2B/BoxLite/another orchestrator or a custom Firecracker runner, only once image-building, kernel updates, jailer config, networking, telemetry and cleanup are automated.

## Gaps
- Daytona OSS runner internals need a code-level verification pass before claiming strong tenant isolation; docs show Docker Compose and network isolation settings, but not a formal isolation/security model for hostile multi-tenant code.
- BoxLite is young; run a spike on target hardware before committing to it for client workloads.
- Final production choice should include threat model, expected concurrency, workload type, egress policy, and whether persistent client state is required.
