# Research: dvassallo/singleserver + Kamal/Tailscale/Cloudflare single-VPS pattern

## Summary
Single Server is a deliberately small PaaS-on-one-box: GitHub push events trigger a local daemon, the daemon builds Docker images on the VPS, Kamal performs healthchecked zero-downtime container swaps, Cloudflare Tunnel publishes app domains, and Tailscale provides private SSH plus a public webhook URL. The pattern is strongest for many small HTTP apps owned by one operator; its main tradeoffs are single-host failure/resource limits, trust in Cloudflare/Tailscale/GitHub, and a simple storage/backup model that needs off-box backup policy for serious data.

## Findings
1. **Architecture: one VPS is the control plane and runtime.** Single Server installs a Go binary that is both systemd daemon and CLI; Docker, Kamal, local git checkouts, config, secrets, persistent storage, backups, and logs all live on the server. App config is an allowlist in `/etc/singleserver/apps.yml`; installing the GitHub App grants repo access, but only listed repos/branches deploy. [Single Server docs](https://github.com/dvassallo/singleserver/blob/main/www/docs/index.html)

2. **Ingress is split by purpose: Tailscale for admin/webhooks, Cloudflare for users.** Tailscale SSH avoids exposing SSH publicly, and Tailscale Funnel exposes the daemon's webhook/setup endpoint as an HTTPS URL for GitHub. Tailscale describes Funnel as a way to route broader-internet traffic to a local service in a tailnet. [Single Server docs](https://github.com/dvassallo/singleserver/blob/main/www/docs/index.html), [Tailscale Funnel](https://tailscale.com/docs/features/tailscale-funnel)

3. **Public app traffic goes through Cloudflare Tunnel, not directly to the VPS.** Single Server creates proxied Cloudflare DNS records and tunnel routes for app domains; `cloudflared` makes outbound connections from the server, and Kamal proxy binds to loopback. Cloudflare documents this as a positive security model where ingress can be blocked and only outbound `cloudflared` traffic allowed. [Single Server docs](https://github.com/dvassallo/singleserver/blob/main/www/docs/index.html), [Cloudflare Tunnel firewall docs](https://developers.cloudflare.com/cloudflare-one/networks/connectors/cloudflare-tunnel/configure-tunnels/tunnel-with-firewall/)

4. **Deploy flow: push -> signed webhook -> allowlist check -> local build -> Kamal rollout.** A push event is verified, checked against repo/branch config, fetched into `/srv/repos/<app>`, built locally with Docker, then deployed by Kamal. First deploy uses `kamal setup`; later deploys use `kamal redeploy`; deploys for the same app serialize, time out after 10 minutes by default, and report GitHub commit status. [Single Server docs](https://github.com/dvassallo/singleserver/blob/main/www/docs/index.html)

5. **Kamal supplies the reliable container swap primitive.** Kamal connects over SSH, installs Docker if needed, builds/pushes/pulls images in normal use, ensures `kamal-proxy` listens on 80/443, starts the new container, waits for a `200 OK` healthcheck, routes traffic to it, stops the old container, and prunes old artifacts. `kamal-proxy` supports host-based routing, automatic HTTPS via Let's Encrypt in direct-to-host setups, healthchecks, path routing, buffering, and per-app proxy config. [Kamal installation docs](https://kamal-deploy.org/docs/installation/), [Kamal proxy docs](https://kamal-deploy.org/docs/configuration/proxy/)

6. **Single Server deliberately avoids an external registry.** Its generated Kamal defaults use `127.0.0.1` as the server, a local registry at `127.0.0.1:5555`, local Docker builder, `ssl: false`, forwarded headers, and loopback-bound proxy; images never leave the box and Cloudflare terminates TLS. [Single Server docs](https://github.com/dvassallo/singleserver/blob/main/www/docs/index.html)

7. **Install/add flow is operator-friendly and mostly idempotent.** On a fresh Ubuntu/Debian/Amazon Linux/Rocky amd64/arm64 box with root access: run `curl -fsSL https://singleserver.com/install.sh | sh`, connect Tailscale, Cloudflare, and GitHub, then `singleserver add https://github.com/you/app`. Installer reruns skip/repair existing pieces; `connect tailscale|cloudflare|github` repairs one integration without disturbing apps. [Single Server README](https://github.com/dvassallo/singleserver), [Single Server docs](https://github.com/dvassallo/singleserver/blob/main/www/docs/index.html)

8. **App contract is intentionally narrow.** Apps must serve HTTP on one port. Secrets are server-side env files, persistent data is an optional host directory mounted at `/storage`, and healthchecks are optional but recommended. Repos can bring their own Dockerfile/Kamal config, or Single Server can generate Dockerfiles for static, Node, or Bun apps. [Single Server docs](https://github.com/dvassallo/singleserver/blob/main/www/docs/index.html)

9. **Safety assumptions are simple but important.** GitHub App permissions are limited to repo contents read, commit statuses write, and push events; webhook signatures are verified; `apps.yml` is the deploy allowlist; destructive cleanup is opt-in (`remove` keeps repos/storage unless flags are passed); failed deploys keep the old container serving because Kamal only switches after healthcheck. [Single Server docs](https://github.com/dvassallo/singleserver/blob/main/www/docs/index.html)

10. **Operational security depends on external identity/tunnel services.** The server can have all inbound firewall traffic blocked after setup, but availability and deployability then depend on Tailscale node key validity, Funnel, Cloudflare Tunnel/DNS, and GitHub App credentials stored under `/etc/singleserver`. Single Server warns that Tailscale node keys expire by default after 180 days and recommends disabling expiry or using tagged auth keys for unattended installs. [Single Server docs](https://github.com/dvassallo/singleserver/blob/main/www/docs/index.html)

11. **Backup/restore is local, app-storage oriented, and SQLite-aware.** `singleserver backup` snapshots an app's storage into timestamped `.tar.gz` files under `/srv/backups/<app>`; SQLite databases are copied through SQLite's backup API for consistency while live. `restore` stops containers, swaps storage from a backup, keeps the old directory as a safety copy, and restarts unless `--no-restart` is used. This is good for simple local recovery, but off-box replication is not built into the documented model. [Single Server docs](https://github.com/dvassallo/singleserver/blob/main/www/docs/index.html)

12. **Limitations: it is not a multi-host PaaS.** The design accepts one-machine blast radius, finite CPU/RAM/disk/build capacity, no scheduler, no horizontal scaling, no tenant isolation beyond Docker/process conventions, and local-state recovery unless the operator exports backups. Kamal itself notes that for multiple servers you need a load balancer; Single Server's defaults are optimized for small containers that boot fast, with repo-level Kamal config as the escape hatch. [Kamal installation docs](https://kamal-deploy.org/docs/installation/), [Single Server docs](https://github.com/dvassallo/singleserver/blob/main/www/docs/index.html)

13. **Transferable ideas for a single-tenant boring-ui deployment platform.** Keep the product as a thin coordinator over standard tools; use an explicit repo/app allowlist; make push/manual deploys share one path; generate conventional Docker/Kamal config but allow escape hatches; run healthchecked zero-downtime swaps; bind origin/proxy privately behind a tunnel; store all state in inspectable files; make doctor/status/logs first-class; make destructive data deletion explicit; and add a first-class off-box backup target if data matters.

## Gaps
- No direct production case study or benchmark for many concurrent Single Server apps beyond the project's claims; validate with a load/build-time test on the target VPS size.
- The docs do not specify encryption/permissions for stored provider tokens and env files; audit file modes and secret handling before adoption.
- Off-box backup/restore, disaster recovery to a new VPS, and Cloudflare/Tailscale/GitHub outage behavior need an explicit boring-ui platform design rather than copying the local backup model as-is.
